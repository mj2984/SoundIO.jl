# SoundIO.jl

SoundIO.jl provides a Julia-native binding to `libsoundio`, merging the low-level control of the original C library with the clarity, safety, and rapid development workflow of Julia.

The goal is to preserve `libsoundio`’s philosophy—minimal abstraction, maximum performance—while taking advantage of Julia’s strengths: type specialization, expressive syntax, automatic memory management, and interactive exploration. The result is a system where you can have the cake and eat it too: C-level performance, Julia-level readability, and the option to dive into raw pointers when needed.

---

## Why SoundIO.jl?

SoundIO.jl gives you **direct, low-level access to audio streams**, which enables:

- **Transparent processing:** See exactly what happens at every stage of the audio pipeline.  
- **Fused operations:** Combine steps like signal processing and data packing efficiently, reducing overhead.  
- **Maintained audio flow:** Preserve a clear, consistent input-to-output path without unexpected shifts or transformations.  
- **Performance-first design:** Specialized callbacks and minimal allocations ensure C-level speed for real-time audio.  
- **Extensible and safe:** While exposing raw pointers, SoundIO.jl wraps them safely and integrates seamlessly with Julia’s type system and tasks.  

In short, it’s designed for developers who want **full visibility, precise control, and high performance** over their audio pipelines.

## 🚀 Key Features

### Zero-Allocation Callbacks

Callback functions are generated by specializing on concrete stream types. This eliminates dynamic dispatch, avoids symbol lookups, and allows C to call Julia with minimal overhead.

### Type-Driven Architecture

All major operations dispatch on strongly typed wrappers (`SoundIOContext`, `SoundIODevice`, `SoundIOOutStream`, etc.). This gives you:

* predictable performance
* readable code
* easy extensibility via multiple dispatch

### Threaded Synchronicity

A custom `AudioCallbackSynchronizer` and `@atomic` flags bridge high-priority C audio threads with Julia tasks. You can run real-time audio loops on pinned Julia threads with deterministic behavior.

### Fast Wait & Event-Driven Wait

Choose between:

* **Fast wait loops** (`wait(context)` / `wait(device)`) for low-latency real-time processing
* **Event-driven waits** triggered by callbacks for GUI-friendly or lower-frequency workflows

### Julia-Native Resource Management

Overloads of `open`, `close`, `isopen`, and `show` make audio devices feel like native Julia I/O resources.

### Rich REPL Inspection

Custom pretty-printing gives immediate insight into contexts, devices, formats, and active streams.

### No Manual Setup

Uses `libsoundio_jll` for cross-platform binary distribution.

## ⚡ Type-Specialized Callbacks

SoundIO.jl generates audio callbacks by specializing on the concrete stream type.

For example, `make_sound_output_callback(stream, userfunc)` produces a closure that:

* is compiled specifically for that stream type
* captures arbitrary Julia state
* exposes safe Julia arrays or raw pointers as needed
* avoids dynamic dispatch inside the callback
* integrates directly into the stream object (`stream.write_callback = …`)

This design gives you C-level performance with Julia-level clarity.
The callback reads like normal Julia code, but executes like optimized C.

This is one of the core architectural advantages of SoundIO.jl.

---

## ⏱️ Synchronization Models

SoundIO.jl exposes both synchronization strategies supported by `libsoundio`:

### Fast Wait

Use `wait(context)` or `wait(device)` inside a tight loop for real-time synthesis or low-latency processing.

Ideal for:

* procedural audio
* live DSP
* tight control loops
* pinned-thread workers

### Event-Driven Wait

Callbacks can trigger Julia tasks at lower frequency, suitable for:

* GUI applications
* asynchronous pipelines
* monitoring or metering
* background processing

Both models are implemented in pure Julia and integrate cleanly with Julia’s scheduler and `Task`s.

## 📦 Julia-Native Resource Management

SoundIO.jl overloads several `Base` methods to make audio devices feel like native Julia resources:

* `open(device)` opens an input or output stream
* `close(stream)` safely releases resources
* `isopen(stream)` checks stream state
* `show` methods provide rich REPL output

This enables idioms like:

```julia
open(device) do stream
    # read or write audio here
end
```

This keeps the API minimal and Julian while preserving full low-level control.

## 🔍 Self-Documenting Interactivity

A core goal of SoundIO.jl is to support code that is self-documenting and expressive.

### Readable Formats

Use symbols like `:Int16Little`, `:Int32Little`, or `:Float32Big` instead of magic numbers.

### Expressive Errors

Hardware issues surface as descriptive symbols such as `:BackendDisconnected` or `:OpeningDeviceFailed`.

### Rich REPL Inspection

Pretty-printed contexts and devices show formats, channel layouts, latency ranges, and active streams.

**Example:**

```julia

      └─ Active Streams: 1
    2. 🎤 [  ] USB Microphone
```

## 🛠 Quick Start: Playback from RAM

SoundIO.jl makes it easy to stream audio directly from a Julia matrix using the high-level `play_audio` method.

```julia
using SoundIO

# 1. Prepare your audio data (Stereo Matrix)
fs = 44100
t = 0:1/fs:1
sine = Int32.(round.(sin.(2π * 440 .* t) .* (2^31 - 1)))
audio_data = vcat(sine', sine') 

# 2. Open a Context and Play
SoundIOContext() do ctx
    enumerate_devices!(ctx)
    
    # Select a 'Raw' device for the lowest latency possible
    device = first(filter(d -> !d.is_input && d.is_raw, ctx.devices))
    
    println("🎶 Playing on: $(device.name)")
    play_audio(audio_data, Int(fs), ctx, device, :Int32Little)
end
```

## ⚡ Advanced: High-Performance Threaded Worker

For real-time synthesis or heavy processing, you can “pin” an audio worker task to a specific CPU thread:

```julia
sync = AudioCallbackSynchronizer()

# Create a task and stick it to Thread 5 for maximum stability
worker_task = @task audio_streamer_ram_playback(sync, audio_data)
ccall(:jl_set_task_tid, Cvoid, (Any, Int16), worker_task, 5) 
worker_task.sticky = true
schedule(worker_task)

# The C callback will now signal Thread 5 whenever the hardware buffer is ready.
```

This pattern is ideal for ultra-low-latency applications.

## 🛡 Safety & Implementation Details

### Reference Counting

`SoundIODevice` automatically manages C-side reference counts to prevent use-after-free errors.

### GC Preservation

Critical buffers are wrapped in `GC.@preserve` during playback to ensure the C thread always has a valid memory address.

### Error Handling

Errors are returned as clean Julia symbols instead of raw integers.

### Safe by Default, Unsafe When Needed

You can work entirely with safe Julia arrays, or drop down to raw pointers for advanced scenarios.

## 🌐 Universal Data Transport

Beyond audio, SoundIO.jl can act as a high-performance synchronous data transport layer.

Using **Exclusive Mode**, the library provides a bit-perfect pipe that bypasses OS mixers—ideal for:

* biomedical signals (ECG/EEG)
* industrial sensor data
* high-precision telemetry
* any application requiring deterministic, low-latency streaming

As long as the hardware is class-compliant, SoundIO.jl can drive it.

---

## 🧱 Extensibility by Design

SoundIO.jl intentionally stays close to `libsoundio`’s minimal philosophy. However, the Julia wrapper architecture makes it easy to build higher-level tools:

* lock-free ring buffers
* DSP pipelines
* audio graphs
* async/Task-based processors
* custom callback generators
* resamplers and format converters

Because everything is implemented using multiple dispatch, extending the system is as simple as defining new methods on existing types.
